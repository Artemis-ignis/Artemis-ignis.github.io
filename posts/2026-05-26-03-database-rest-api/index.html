2026-05-26 강의자료 03: 데이터베이스와 REST API

이 글은 [수업자료] 데이터베이스와 REST API.pdf를 보고 다시 정리한 복습 기록이다. 슬라이드를 그대로 옮기기보다는, 나중에 봤을 때 “아, 이게 그 말이었지” 하고 떠올릴 수 있게 쉬운 말과 예시 중심으로 적었다.

한 줄 요약

클라이언트는 API로 서버에 요청하고, 서버는 데이터베이스에 쿼리를 보내 필요한 데이터를 읽거나 바꾼 뒤 응답을 돌려준다.

수업 흐름

REST API는 웹에서 프로그램끼리 대화하는 약속이다. 주소는 대상이 되고, GET, POST, PATCH, DELETE 같은 요청 방식은 행동이 된다. 서버는 요청을 받은 뒤 필요한 검사를 하고 데이터베이스에 쿼리를 보낸다. 데이터는 보통 JSON으로 주고받지만, 공공데이터나 오래된 시스템에서는 XML도 자주 만난다.

쉽게 이해하기

게시판을 예로 들면 GET /posts는 글 목록을 보여 달라는 요청이고, POST /posts는 새 글을 저장해 달라는 요청이다. PATCH /posts/1은 1번 글을 고치는 요청, DELETE /posts/1은 1번 글을 삭제하는 요청이다. 주소와 동사를 같이 읽으면 API가 무슨 일을 하는지 훨씬 덜 헷갈린다.

헷갈렸던 단어

용어 내가 이해한 뜻
API 프로그램끼리 데이터를 주고받기 위한 약속된 통로
REST 주소와 HTTP 요청 방식을 이용해 자원을 다루는 API 스타일
쿼리 데이터베이스에 보내는 질문이나 요청
CRUD Create, Read, Update, Delete의 네 가지 기본 데이터 작업

짧은 예시

회원 1이 게시글을 작성하면 브라우저는 서버에 POST 요청을 보낸다. 서버는 로그인 여부를 확인하고, 글 제목과 내용을 검사한 뒤, 데이터베이스에 INSERT 쿼리를 실행한다. 성공하면 서버는 ‘저장됨’이라는 응답을 보내고, 화면은 새 글 페이지로 이동한다.

복습 체크

  • GET, POST, PATCH, DELETE의 차이를 게시판 예시로 말할 수 있는가?
  • JSON과 XML이 데이터 표현 형식이라는 점을 구분할 수 있는가?
  • 클라이언트, 서버, 데이터베이스가 어떤 순서로 연결되는지 그릴 수 있는가?

오늘의 정리

REST API와 데이터베이스는 웹서비스의 뼈대에 가깝다. 화면에서 버튼 하나를 눌렀을 뿐인데, 뒤에서는 요청, 검증, 쿼리, 응답이 이어진다. 이 흐름을 이해하면 백엔드가 왜 필요한지 자연스럽게 보인다.
