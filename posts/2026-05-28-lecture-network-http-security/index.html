네트워크는 컴퓨터끼리 대화하는 길이다

네트워크는 여러 컴퓨터가 서로 데이터를 주고받을 수 있게 연결된 구조다. 우리가 웹사이트 주소를 입력하면 내 브라우저는 서버를 찾아가고, 서버는 HTML, CSS, 이미지, JSON 같은 데이터를 다시 보내준다.

이때 IP 주소는 집 주소에 가깝다. DNS는 사람이 읽기 쉬운 도메인 이름을 실제 IP 주소로 바꿔주는 전화번호부 같은 역할을 한다.

HTTP는 웹의 대화 규칙이다

HTTP는 웹에서 클라이언트와 서버가 대화하는 규칙이다. 브라우저가 서버에 요청을 보내고, 서버는 상태 코드와 함께 응답을 돌려준다.

상태 코드 의미
200 요청 성공
301/302 다른 주소로 이동
400 잘못된 요청
401 인증 필요
403 권한 없음
404 찾을 수 없음
500 서버 내부 오류
503 서버가 일시적으로 처리 불가

개발할 때 404는 주소나 라우팅 문제를 의심하고, 500은 서버 코드나 설정 문제를 의심한다. 상태 코드를 읽을 수 있으면 오류 상황을 더 빠르게 좁힐 수 있다.

주요 프로토콜

프로토콜은 통신 약속이다. 같은 언어와 규칙을 써야 서로 이해할 수 있다.

프로토콜 역할
IP 목적지 주소를 기준으로 패킷 전달
DNS 도메인을 IP 주소로 변환
DHCP 네트워크에 접속한 기기에 IP 자동 할당
TCP 신뢰성 있는 연결 중심 통신
UDP 빠른 전송이 중요한 통신
FTP 파일 전송
SMTP 이메일 전송
SSH 원격 서버 접속
SSL/TLS 암호화된 안전한 통신

웹사이트 주소가 https://로 시작하면 SSL/TLS로 암호화된 통신을 사용한다. 로그인, 결제, 개인정보 입력 페이지에서 HTTPS가 중요한 이유다.

쿠키, 세션, 토큰, JWT

HTTP는 기본적으로 이전 요청을 기억하지 않는다. 그래서 로그인 상태 같은 정보를 유지하려면 별도의 방법이 필요하다.

쿠키는 브라우저에 저장되는 작은 정보다. 세션은 서버가 로그인 상태를 기억하는 방식이다. 토큰은 클라이언트가 인증 정보를 들고 다니는 방식이고, JWT는 토큰 안에 정보를 담는 형식 중 하나다.

보안에서 중요한 것은 “누구인지 확인하는 것"과 “무엇을 할 권한이 있는지 확인하는 것"을 분리해서 생각하는 것이다. 로그인은 인증이고, 관리자 페이지 접근 가능 여부는 권한 검사다.

캐시와 CORS

캐시는 자주 쓰는 데이터를 임시로 저장해 빠르게 보여주는 방법이다. 이미지를 매번 새로 받지 않고 저장해두면 사이트가 빨라진다. 다만 오래된 데이터가 남을 수 있으므로 언제 새로 받을지 규칙이 필요하다.

CORS는 다른 출처의 리소스 요청을 제어하는 브라우저 보안 정책이다. 프론트엔드와 백엔드 주소가 다를 때 CORS 오류를 자주 만난다. 이때 서버가 어떤 출처를 허용할지 정확히 설정해야 한다.

오늘의 정리

네트워크 수업은 웹서비스의 보이지 않는 도로망을 이해하는 시간이었다. API 요청 하나도 IP, DNS, HTTP, 상태 코드, 보안 정책을 거쳐 움직인다. 개발자는 화면만 보는 것이 아니라 요청이 어디로 가고, 어떤 규칙으로 응답이 오는지 읽을 수 있어야 한다.
