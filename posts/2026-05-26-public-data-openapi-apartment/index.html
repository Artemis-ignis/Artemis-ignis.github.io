2026-05-26 이어드림스쿨 수업 정리: 공공데이터 OpenAPI 읽는 법

아파트 매매 실거래가 자료 기술문서는 오늘 배운 API 개념을 실제 문서에서 확인하는 예시로 보기 좋았다. 수업에서 배운 REST API, 요청과 응답, XML, 인증키가 한꺼번에 나온다.

중요한 점부터 정리하면, 공공데이터 API는 주소만 브라우저에 넣는다고 끝나는 게 아니다. 필요한 파라미터를 같이 보내야 하고, 인증키도 안전하게 다뤄야 한다.

OpenAPI는 공개된 약속이다

OpenAPI는 외부에서 사용할 수 있도록 공개된 API다. 공공데이터 API는 정부나 기관이 가진 데이터를 정해진 방식으로 요청할 수 있게 해준다.

하지만 “공개"라는 말이 아무렇게나 써도 된다는 뜻은 아니다. 요청 주소, 필수 파라미터, 인증 방식, 응답 형식을 문서대로 맞춰야 한다.

필수 파라미터를 확인해야 한다

아파트 매매 실거래가 API 같은 공공데이터 API는 보통 요청할 때 필요한 값이 있다.

예를 들어 이런 값이 필요할 수 있다.

파라미터 의미
serviceKey API 인증키
지역코드 조회할 지역
계약월 조회할 계약 시점

실제 serviceKey 값은 절대 블로그 글이나 공개 저장소에 넣으면 안 된다. 인증키는 비밀번호처럼 다뤄야 한다.

요청 모양은 대략 이렇게 이해하면 된다.

GET 공공데이터_API_주소?serviceKey=발급받은키&LAWD_CD=지역코드&DEAL_YMD=계약월

위 예시는 구조를 설명하기 위한 자리 표시자다. 실제 키 값은 넣지 않았다.

REST GET 요청과 XML 응답

해당 API는 REST GET 방식으로 요청한다. 브라우저 주소창에 URL을 넣는 것도 GET 요청의 한 종류라고 볼 수 있다.

응답은 XML 형식으로 받을 수 있다. XML은 태그로 데이터를 감싸는 형식이다.

<item>
  <region>지역 A</region>
  <dealMonth>계약월</dealMonth>
  <amount>거래금액</amount>
</item>

실제 문서의 필드명은 문서 기준으로 확인해야 한다. 블로그 글에서는 개인정보나 인증키가 들어가지 않도록 익명 예시만 사용했다.

API 문서를 읽을 때 보는 순서

처음 API 문서를 보면 표와 항목이 많아서 복잡하다. 나는 다음 순서로 보는 게 편했다.

  1. 요청 주소가 무엇인지 확인한다.
  2. 요청 방식이 GET인지 POST인지 확인한다.
  3. 필수 파라미터를 확인한다.
  4. 인증키가 필요한지 확인한다.
  5. 응답 형식이 JSON인지 XML인지 확인한다.
  6. 응답 필드 중 실제로 쓸 값만 표시해본다.
  7. 오류 코드나 제한 사항을 확인한다.

이 순서로 보면 문서가 조금 덜 무섭다.

서비스에 붙일 때는 서버 쪽에서 다루는 게 안전하다

공공데이터 API를 웹서비스에 붙일 때는 인증키를 프론트엔드에 그대로 노출하지 않는 편이 안전하다. 브라우저에서 보이는 코드에 키를 넣으면 다른 사람이 가져갈 수 있기 때문이다.

보통은 서버가 공공데이터 API에 요청하고, 서버가 필요한 데이터만 정리해서 클라이언트에 보내는 구조가 낫다.

클라이언트 → 내 서버 → 공공데이터 API → 내 서버 → 클라이언트

이렇게 하면 인증키를 서버 쪽에서 관리할 수 있고, 응답 데이터도 서비스에 맞게 정리할 수 있다.

내 식으로 정리

공공데이터 OpenAPI 문서는 오늘 배운 개념을 실제로 확인하는 연습장 같았다. API 주소, GET 요청, 파라미터, 인증키, XML 응답이 모두 들어 있다.

핵심은 문서대로 요청을 만드는 것과 인증키를 안전하게 다루는 것이다. API를 쓸 줄 안다는 건 주소를 복사하는 게 아니라, 필요한 값과 응답 구조를 읽고 서비스 흐름에 맞게 연결하는 일이다.
