2026-05-26 이어드림스쿨 수업 정리: REST API와 데이터베이스 기초

데이터베이스와 REST API 자료는 오늘 내용 중 가장 서비스 개발에 가까운 파트였다. 웹서비스가 화면과 서버의 대화라면, API는 그 대화의 약속이고 데이터베이스는 기억을 저장하는 공간이다.

처음엔 용어가 많아서 복잡해 보이지만, 큰 흐름은 이렇다.

클라이언트가 API로 요청한다 → 서버가 비즈니스 로직을 처리한다 → 서버가 DB에 쿼리한다 → 결과를 응답한다

API와 REST API

API는 프로그램끼리 데이터를 주고받는 약속된 통로다. 사람이 메뉴판을 보고 주문하듯이, 프로그램도 API라는 약속을 보고 요청한다.

REST API는 웹에서 많이 쓰는 API 방식이다. 주소와 요청 방식으로 데이터를 다룬다.

방식 뜻 게시판 예시
GET 가져오기 GET /posts 글 목록 조회
POST 새로 만들기 POST /posts 새 글 작성
PATCH 수정하기 PATCH /posts/1 1번 글 수정
DELETE 삭제하기 DELETE /posts/1 1번 글 삭제

주소는 “무엇을 다룰지"를 나타내고, 요청 방식은 “그것으로 무엇을 할지"를 나타낸다.

JSON과 XML

JSON과 XML은 데이터를 주고받는 형식이다.

JSON은 요즘 웹 API에서 많이 쓰인다. 중괄호를 쓰고, 구조가 비교적 간단하다.

{
  "name": "사용자 A",
  "age": 20
}

XML은 태그를 쓰는 형식이다. 공공기관 API, 오래된 시스템, RSS 같은 곳에서 볼 수 있다.

<user>
  <name>사용자 A</name>
  <age>20</age>
</user>

둘 다 데이터를 담는 방식이다. JSON은 간단한 상자 느낌이고, XML은 이름표가 붙은 봉투 느낌이다.

동기와 비동기

동기는 작업이 끝날 때까지 기다리는 방식이다. 앞의 일이 끝나야 다음 일을 할 수 있다.

비동기는 작업을 맡겨두고 다른 일을 할 수 있는 방식이다. 다운로드를 백그라운드에서 돌리면서 다른 작업을 계속할 수 있다면 비동기 느낌이다.

게임으로 보면 캐스팅이 끝날 때까지 내 캐릭터가 다음 행동을 못 하는 건 동기와 비슷하다. 반대로 주변 유저들은 계속 움직이고, 다른 이벤트도 동시에 진행되는 상황은 비동기 감각에 가깝다.

웹서비스에서는 서버 응답을 기다리는 동안 화면 전체가 멈추면 답답하다. 그래서 비동기 처리가 자주 쓰인다.

데이터베이스 종류

데이터베이스는 하나만 있는 게 아니다. 저장하려는 데이터 성격에 따라 선택이 달라진다.

종류 쉽게 말하면 잘 맞는 데이터 예시
RDB 엑셀 표처럼 정리된 DB 회원, 주문, 결제, 게시글 MySQL, Oracle, PostgreSQL
NoSQL 자유로운 형태의 저장소 채팅, 로그, 문서, 검색 MongoDB, Elasticsearch, Redis
VectorDB 의미가 비슷한 데이터를 찾는 DB AI 검색, RAG, 문서 검색 ChromaDB

RDB는 정확한 데이터에 강하다. 회원가입, 주문, 결제처럼 틀이 분명한 데이터에 잘 맞는다.

NoSQL은 틀이 자주 바뀌거나 데이터 형태가 자유로운 경우에 좋다. 로그나 채팅처럼 양이 많고 형식이 조금씩 다른 데이터에 쓰기 쉽다.

VectorDB는 AI 쪽에서 자주 나온다. 질문과 문서의 문장이 완전히 같지 않아도 의미가 비슷하면 찾아주는 방식이다.

Firebase와 Supabase

Firebase와 Supabase는 단순한 DB라기보다 백엔드 기능 묶음에 가깝다.

Firebase는 Google이 만든 앱·웹 백엔드 도구 세트다. DB, 인증, 파일 저장, 함수, 호스팅, 알림 등을 제공한다. 앱 중심 MVP를 빠르게 만들 때 편하다.

Supabase는 PostgreSQL 기반 백엔드 도구 세트다. DB, 인증, 파일 저장, 자동 API, 실시간 기능을 제공한다. SQL 기반 웹서비스와 관리자 페이지 구조를 이해하기 좋다.

둘 다 무료로 시작할 수 있지만 사용량이 늘면 유료가 될 수 있다. 작은 프로젝트에서는 무료 범위로 시작하되, 실제 운영 전에는 요금과 제한을 확인해야 한다.

쿼리와 CRUD

쿼리는 데이터베이스에 하는 질문이나 요청이다. 쿼리문은 그 질문을 SQL 문법으로 쓴 문장이다.

SELECT * FROM users;

“회원 목록 보여줘"가 쿼리라면, 위 SQL은 그 쿼리를 데이터베이스가 알아듣는 문장으로 쓴 것이다.

CRUD는 데이터베이스에서 데이터를 다루는 가장 기본적인 네 가지 행동이다.

CRUD 뜻 SQL 게시판 예시
Create 만들기 INSERT 글 작성
Read 읽기 SELECT 글 보기
Update 수정하기 UPDATE 글 수정
Delete 삭제하기 DELETE 글 삭제

게시판, 쇼핑몰, 관리자 페이지 대부분은 결국 CRUD를 바탕으로 만들어진다.

분석가와 개발자의 SQL

데이터 분석가도 SQL을 알면 좋고, 개발자는 SQL이 필수에 가깝다.

분석가는 데이터를 꺼내고 해석하기 위해 SQL을 쓴다. 개발자는 회원가입, 로그인, 주문, 게시글, 관리자 페이지 같은 실제 기능을 만들기 위해 SQL을 쓴다.

공부 순서는 이렇게 잡으면 부담이 덜하다.

  1. 기본 문법
  2. JOIN
  3. 서브쿼리
  4. 윈도우 함수
  5. 인덱스
  6. 트랜잭션

처음부터 어려운 기능을 다 외우기보다 SELECT, WHERE, GROUP BY, ORDER BY부터 익히는 게 현실적이다.

인덱스와 조인

인덱스는 데이터베이스에서 데이터를 빨리 찾기 위한 찾아보기다. 책에서 원하는 단어를 찾을 때 처음부터 끝까지 읽지 않고 색인을 보는 것과 비슷하다.

B-Tree와 B+Tree는 범위 검색과 정렬에 강하다. 해시 인덱스는 정확히 같은 값을 빠르게 찾는 데 강하다.

조인은 여러 표를 연결해서 하나의 결과를 만드는 것이다. 회원 표와 주문 표를 연결하면 어떤 회원이 어떤 주문을 했는지 확인할 수 있다.

조인은 편리하지만 데이터가 많아지면 느려질 수 있다. 그래서 인덱스와 쿼리 설계가 중요하다.

서버 오류와 트래픽 과부하

서버 오류는 서버가 요청을 제대로 처리하지 못하는 상태다. 트래픽 과부하는 요청이 너무 많이 몰려서 서버가 버티지 못하는 상태다.

식당으로 보면 트래픽 과부하는 손님이 갑자기 너무 많이 들어온 상황이고, 서버 오류는 주방이 주문을 제대로 처리하지 못하는 상황이다.

오류 의미
500 서버 내부 오류
502 중간 서버가 잘못된 응답을 받음
503 서버가 일시적으로 처리 불가
504 응답 시간이 너무 오래 걸림

해결 방법으로는 서버 확장, 로드밸런서, 캐시, DB 인덱스, 이미지 최적화, 비동기 처리, 대기열, 요청 제한, 모니터링이 있다.

내 식으로 정리

API와 데이터베이스는 따로 배우는 개념처럼 보이지만 실제 서비스에서는 붙어 있다. 사용자가 누르는 버튼 하나가 API 요청이 되고, 그 요청은 서버 로직을 거쳐 데이터베이스 쿼리로 이어진다.

오늘 배운 내용을 제대로 이해하면 “화면은 보이는데 데이터는 어디서 오지?“라는 질문에 답할 수 있다.
